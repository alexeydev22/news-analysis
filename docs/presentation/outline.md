## Слайд 1. Титульный

Заголовок:

> Разработка автоматической диалоговой системы на основе языковой модели для
> анализа экономических новостей

Добавить:

- ФИО студента;
- группа;
- руководитель;
- организация;
- город и год.

Визуал: нейтральный фон, крупная тема, без перегруза.

## Слайд 2. Актуальность

Ключевые тезисы:

- экономические новости быстро влияют на ожидания рынка;
- пользователю сложно вручную сопоставлять много источников;
- нужен инструмент, который отвечает на вопрос и показывает, на какие новости
  он опирается;
- диалоговый формат снижает порог входа.

Визуал: схема «много новостей -> вопрос пользователя -> краткий ответ с
источниками».

## Слайд 3. Цель и задачи

Цель:

> Разработать локальную диалоговую систему, которая находит релевантные
> экономические новости, анализирует их влияние и формирует ответ на вопрос
> пользователя.

Задачи удобно сгруппировать по исследовательской и инженерной частям проекта.

ML/NLP-задачи:

1. Определить текстовое представление новостей.
2. Реализовать retrieval/RAG по источникам.
3. Реализовать классификацию влияния новости.
4. Поддержать template и LLM-режим ответа.

Backend-задачи:

1. Спроектировать микросервисную DDD-архитектуру.
2. Реализовать асинхронные сервисы и фоновые задачи.
3. Сделать React UI с SSE timeline.
4. Подготовить Docker Compose demo-сценарий.

## Слайд 4. Общая архитектура

Показать компоненты:

- `frontend-web`;
- `api-gateway`;
- `news-service`;
- `news-worker`;
- `retrieval-service`;
- `analysis-service`;
- `dialog-service`;
- Redis;
- Qdrant;
- MLflow.

Главная мысль:

> Каждый сервис решает одну задачу, а `api-gateway` связывает их в единый
> диалоговый сценарий.

Акцент: архитектура нужна не сама по себе, а чтобы отделить ML/NLP-компоненты
от инфраструктуры и сделать их заменяемыми.

Визуал: диаграмма из [docs/architecture.md](../architecture.md).

## Слайд 5. Pipeline обработки вопроса

Показать последовательность:

1. Пользователь задает вопрос.
2. Gateway запускает SSE-сценарий.
3. Retrieval-service ищет релевантные новости.
4. Analysis-service оценивает влияние каждой новости.
5. Dialog-service формирует ответ.
6. UI показывает ответ, источники и ход обработки.

Визуал: горизонтальная sequence-схема.

Важно проговорить две линии:

- ML/NLP: retrieval, score источников, классификация влияния, генерация ответа;
- Backend: gateway, контракты сервисов, SSE-события, фоновые задачи.

## Слайд 6. Методы и технологии

Таблица:

| Часть | Выбор |
| --- | --- |
| ML: текстовые признаки | TF-IDF, embeddings |
| ML: классификация | Logistic Regression, tiny transformer mode |
| NLP: retrieval/RAG | Qdrant + контекст источников |
| NLP: генерация | Template fallback + LLM adapter |
| ML tracking | MLflow |
| Backend | FastAPI, asyncio, Granian, Zapros |
| Архитектура | DDD, слои, Protocol, Dishka |
| Очереди и события | Taskiq, Redis, FastStream, SSE |
| UI и запуск | React, Docker Compose, Justfile |

Главная мысль:

> Стек современный, но локальный и не перегруженный лишней инфраструктурой.

## Слайд 7. ML/NLP-компоненты

Показать:

- классы влияния: positive, negative, neutral;
- `tfidf-logreg` как объяснимый baseline;
- выход модели: `impact`, `confidence`, `explanation`;
- retrieval через Qdrant и top-k источники;
- `template` generator как стабильный fallback;
- LLM-режим через OpenAI-compatible локальный сервер;
- выбранная легкая модель: `Qwen3-0.6B-Instruct-GGUF`;
- MLflow как точка учета экспериментов и артефактов.

Формулировка:

> Для защиты используется воспроизводимый режим, а архитектура позволяет
> подключить локальную языковую модель без изменения основного сценария.

## Слайд 11. Заключение

Итоги разделить на две части.

ML/NLP-результат:

- поиск релевантного контекста;
- классификация влияния новостей;
- ответ по источникам с LLM-адаптером;
- MLflow для дальнейших экспериментов.

Backend-результат:

- микросервисы со слоистой DDD-структурой;
- асинхронные API, фоновые задачи и SSE;
- React UI и Docker Compose запуск;
- smoke-сценарий для проверки стенда.

Развитие:

- подключить реальные новостные API;
- добавить историю диалогов в PostgreSQL;
- подключить обученные артефакты дополнительных моделей;
- сравнить качество template и LLM-режимов.
