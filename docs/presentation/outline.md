Ориентир по критериям с изображения:

| Критерий | Баллы | Как закрываем |
| --- | ---: | --- |
| Титул | 1 | Отдельный титульный слайд с темой, ФИО, группой и руководителем. |

## Слайд 1. Титульный

Заголовок:

> Разработка автоматической диалоговой системы на основе языковой модели для
> анализа экономических новостей

Добавить:

- ФИО студента;
- группа;
- руководитель;
- организация;
- город и год.

Визуал: нейтральный фон, крупная тема, без перегруза.

## Слайд 2. Актуальность

Ключевые тезисы:

- экономические новости быстро влияют на ожидания рынка;
- пользователю сложно вручную сопоставлять много источников;
- нужен инструмент, который отвечает на вопрос и показывает, на какие новости
  он опирается;
- диалоговый формат снижает порог входа.

Визуал: схема «много новостей -> вопрос пользователя -> краткий ответ с
источниками».

## Слайд 3. Цель и задачи

Цель:

> Разработать локальную диалоговую систему, которая находит релевантные
> экономические новости, анализирует их влияние и формирует ответ на вопрос
> пользователя.

Задачи:

1. Спроектировать микросервисную архитектуру.
2. Реализовать загрузку и индексацию новостей.
3. Реализовать векторный поиск релевантных источников.
4. Реализовать анализ влияния новости.
5. Реализовать генерацию ответа с поддержкой языковой модели.
6. Подготовить web UI и demo-сценарий.

## Слайд 4. Общая архитектура

Показать компоненты:

- `frontend-web`;
- `api-gateway`;
- `news-service`;
- `news-worker`;
- `retrieval-service`;
- `analysis-service`;
- `dialog-service`;
- Redis;
- Qdrant;
- MLflow.

Главная мысль:

> Каждый сервис решает одну задачу, а `api-gateway` связывает их в единый
> диалоговый сценарий.

Визуал: диаграмма из [docs/architecture.md](../architecture.md).

## Слайд 5. Pipeline обработки вопроса

Показать последовательность:

1. Пользователь задает вопрос.
2. Gateway запускает SSE-сценарий.
3. Retrieval-service ищет релевантные новости.
4. Analysis-service оценивает влияние каждой новости.
5. Dialog-service формирует ответ.
6. UI показывает ответ, источники и ход обработки.

Визуал: горизонтальная sequence-схема.

## Слайд 6. Методы и технологии

Таблица:

## Слайд 7. Реализация анализа и генерации

Показать:

- `tfidf-logreg` как стабильный режим анализа для demo;
- предусмотренные режимы `embedding-logreg` и `tiny-transformer-classifier`;
- `template` generator как стабильный fallback;
- LLM-режим через OpenAI-compatible локальный сервер;
- выбранная легкая модель: `Qwen3-0.6B-Instruct-GGUF`.

Формулировка:

> Для защиты используется воспроизводимый режим, а архитектура позволяет
> подключить локальную языковую модель без изменения основного сценария.

## Требования к оформлению

- 10-12 слайдов достаточно.
- На слайде не больше 5-6 коротких тезисов.
- Схемы и скриншоты важнее длинного текста.
- Все подписи на русском языке.
- Единый шрифт и единая цветовая схема.
- Код показывать только фрагментами, если он нужен для объяснения архитектуры.
