# Заметки докладчика

Ориентир: 5-7 минут. Текст не нужно читать дословно, он нужен как основа для
устного выступления.

## Слайд 1. Титульный

Здравствуйте. Тема моей курсовой работы: «Разработка автоматической диалоговой
системы на основе языковой модели для анализа экономических новостей».

В работе я реализовал локальное приложение, которое позволяет задать вопрос по
экономическим новостям, найти релевантные источники, оценить влияние этих
источников и получить итоговый ответ в интерфейсе чата.

## Слайд 2. Актуальность

Экономические новости появляются постоянно и могут влиять на ожидания рынка:
например, новости о ВВП, инфляции, ставке центрального банка, безработице или
экспорте.

Проблема в том, что пользователю сложно вручную анализировать много источников
и быстро понимать общий смысл. Поэтому актуальна диалоговая система, которая
принимает вопрос естественным языком и возвращает краткий ответ с указанием
источников.

## Слайд 3. Цель и задачи

Цель работы - разработать локальную диалоговую систему, которая находит
релевантные экономические новости, анализирует их влияние и формирует ответ на
вопрос пользователя.

Для этого были решены задачи проектирования архитектуры, реализации сервисов
загрузки, поиска, анализа и генерации ответа, создания web-интерфейса и
подготовки воспроизводимого demo-сценария.

## Слайд 4. Архитектура

Система сделана как микросервисный стенд. В центре находится `api-gateway`,
который принимает запрос от frontend и оркестрирует весь сценарий.

Отдельно выделены `news-service` для работы с CSV dataset, `retrieval-service`
для поиска источников, `analysis-service` для оценки влияния и `dialog-service`
для генерации ответа. Для фоновых задач используется `news-worker`, Taskiq и
Redis. Для векторного поиска используется Qdrant.

Такое разделение делает систему поддерживаемой: каждый сервис отвечает за свою
часть процесса.

## Слайд 5. Pipeline обработки вопроса

Когда пользователь задает вопрос, frontend отправляет запрос в `api-gateway`.
Gateway отдает события через SSE, поэтому пользователь видит ход обработки.

Сначала система ищет релевантные новости в `retrieval-service`. Затем каждая
найденная новость передается в `analysis-service`, где определяется ее влияние:
позитивное, негативное или нейтральное. После этого `dialog-service` формирует
итоговый ответ на основе вопроса и найденных источников.

## Слайд 6. Методы и технологии

Backend написан на FastAPI с асинхронным подходом и запускается через Granian.
Архитектура сервисов построена по слоям и DDD: есть доменный слой, application
use cases, infrastructure adapters, presentation layer и main layer.

Для dependency injection используется Dishka, для typed contracts - Pydantic,
для межсервисных вызовов - Zapros. Фоновые задачи реализованы через Taskiq и
Redis, события - через FastStream. Для векторного поиска выбран Qdrant, для
ML-экспериментов - MLflow. Интерфейс реализован на React.

## Слайд 7. Анализ и генерация

Для анализа влияния новости основной demo-режим - `tfidf-logreg`. Он выбран
потому, что быстро запускается локально и стабильно работает в Docker Compose.

Также в системе предусмотрены режимы `embedding-logreg` и
`tiny-transformer-classifier`, которые можно подключать через артефакты моделей.

Для генерации ответа `dialog-service` поддерживает два режима: template fallback
и LLM-режим. LLM-режим работает через OpenAI-compatible локальный сервер,
например `llama.cpp`, с легкой моделью `Qwen3-0.6B-Instruct-GGUF`.

## Слайд 8. Интерфейс

В интерфейсе пользователь может выбрать модель анализа, посмотреть CSV dataset,
запустить индексацию и задать вопрос. После ответа справа отображаются
источники, а в центре - результат и timeline обработки.

Это важно, потому что система не просто генерирует текст, а показывает, какие
источники были найдены и как они были оценены.

## Слайд 9. Результаты проверки

Для проверки подготовлен demo-сценарий. Стенд запускается командой
`just demo-up`, а автоматическая проверка выполняется командой
`just demo-smoke`.

Smoke-сценарий проверяет health endpoints, preview CSV, индексацию новостей,
постановку фоновой задачи, SSE-поток чата и доступность frontend.

Ручная проверка выполняется через вопрос: «Что означает рост ВВП и снижение
инфляции для рынка?». Система возвращает ответ и список источников.

## Слайд 10. Соответствие теме

Проект соответствует теме работы по нескольким пунктам. Во-первых, есть
автоматический диалоговый интерфейс. Во-вторых, есть анализ экономических
новостей. В-третьих, есть retrieval pipeline и генерация ответа на основе
найденного контекста. В-четвертых, архитектура содержит LLM-адаптер, то есть
может работать с локальной языковой моделью.

## Слайд 11. Заключение

В результате была разработана микросервисная диалоговая система для анализа
экономических новостей. Она запускается локально, имеет web-интерфейс,
индексирует demo dataset, ищет релевантные новости, анализирует их влияние и
формирует ответ.

Дальше систему можно развивать: подключить реальные новостные API, сохранять
историю диалогов в PostgreSQL, обучить дополнительные модели анализа и сравнить
качество template и LLM-режимов.

## Короткие ответы на возможные вопросы

**Почему микросервисы, если проект учебный?**

Потому что тема включает несколько разных задач: загрузку данных, поиск,
анализ, генерацию ответа и интерфейс. Микросервисы позволяют показать эти
границы явно, но код внутри сервисов остается достаточно простым.

**Почему Redis, а не RabbitMQ?**

Redis легче для локального стенда и закрывает две задачи: очередь Taskiq и канал
событий FastStream.

**Почему template-режим, если тема про языковую модель?**

Template-режим нужен для стабильной защиты без зависимости от локальной модели.
При этом в архитектуре реализован LLM-адаптер, который подключает
OpenAI-compatible сервер с локальной языковой моделью.

**Почему `tfidf-logreg` выбран для demo?**

Он быстрый, воспроизводимый и не требует скачивания тяжелых ML-артефактов.
Дополнительные режимы предусмотрены архитектурно и могут быть подключены через
артефакты моделей.

**Что является главным результатом?**

Работающий end-to-end сценарий: CSV новости индексируются, пользователь задает
вопрос, система находит источники, анализирует их и формирует ответ.
