# Структура курсовой работы

Тема: «Разработка автоматической диалоговой системы на основе языковой модели
для анализа экономических новостей».

Документ задает рабочую структуру пояснительной записки. Его можно использовать
как оглавление перед переносом текста в `.docx`.

## Введение

Что раскрыть:

- актуальность автоматического анализа экономических новостей;
- проблему информационной перегрузки пользователей;
- объект исследования: экономические новости и их влияние;
- предмет исследования: методы поиска, анализа и генерации ответа в
  диалоговой системе;
- цель работы;
- задачи работы;
- практическую значимость.

Рабочая цель:

> Разработать локальную автоматическую диалоговую систему, которая принимает
> вопрос пользователя, находит релевантные экономические новости, анализирует
> их влияние и формирует ответ на основе найденных источников.

Рабочие задачи:

1. Изучить подходы к построению диалоговых систем и анализу текстовых новостей.
2. Определить ML/NLP pipeline: текстовое представление, retrieval,
   классификацию влияния и генерацию ответа.
3. Спроектировать микросервисную архитектуру системы.
4. Реализовать backend-сервисы для поиска, анализа, генерации ответа и загрузки
   новостей.
5. Реализовать веб-интерфейс для диалогового сценария.
6. Подготовить воспроизводимый demo dataset и сценарий проверки.
7. Проверить работоспособность системы через автоматический smoke-сценарий и
   ручную демонстрацию.

## Глава 1. Анализ предметной области

Содержание главы:

- экономические новости как источник данных для принятия решений;
- типовые задачи NLP для новостей: классификация тональности, классификация
  влияния, поиск похожих документов, вопросно-ответные системы;
- понятие RAG-подхода: поиск релевантного контекста перед генерацией ответа;
- роль языковой модели в диалоговой системе;
- требования к локальному учебному стенду: воспроизводимость, умеренные
  ресурсы, возможность запуска через Docker Compose.

Ожидаемый вывод главы:

> Для курсовой работы подходит гибридный подход: векторный поиск выбирает
> релевантные новости, модель анализа оценивает влияние каждой новости, а
> диалоговый сервис формирует итоговый ответ на основе найденного контекста.

## Глава 2. Проектирование диалоговой системы

Содержание главы:

- общая архитектура проекта;
- перечень микросервисов и их ответственность;
- слоистая архитектура backend-сервисов;
- DDD-разделение на `domain`, `application`, `infrastructure`,
  `presentation`, `main`;
- использование `Protocol` как интерфейсов портов;
- dependency injection через Dishka;
- выбор инфраструктуры: Redis, Qdrant, PostgreSQL, MLflow;
- сценарии взаимодействия сервисов.

Диаграммы:

- схема микросервисов из [docs/architecture.md](../architecture.md);
- sequence diagram обработки вопроса;
- схема слоев одного backend-сервиса.

## Глава 3. Реализация программного проекта

Содержание главы:

- структура monorepo;
- общий framework-пакет;
- contracts-пакет с DTO;
- `api-gateway` как единая точка входа;
- ML/NLP pipeline: TF-IDF, embeddings, retrieval и классификация влияния;
- `retrieval-service` и Qdrant;
- `analysis-service`, режим `tfidf-logreg` и расширяемые режимы моделей;
- `dialog-service` с template fallback и LLM-адаптером;
- `news-service` и `news-worker`;
- React UI;
- Docker Compose, Dockerfiles и Justfile.

Важно подчеркнуть:

- код не сводится к одному монолитному скрипту;
- сервисы имеют понятные границы;
- demo-режим остается легким и воспроизводимым;
- LLM-режим подключается через OpenAI-compatible локальный сервер.

## Заключение

Что отразить:

- цель достигнута: система принимает вопрос и строит ответ по экономическим
  новостям;
- реализованы ML/NLP-компоненты: retrieval, анализ влияния, confidence score,
  объяснение результата и LLM-адаптер;
- реализованы backend-компоненты: микросервисы, DDD, фоновые задачи, SSE и UI;
- подготовлен локальный воспроизводимый стенд;
- архитектура допускает замену моделей, хранилищ и источников данных;
- возможные направления развития.

Направления развития:

- подключение реальных новостных API;
- обучение и подключение `embedding-logreg` и `tiny-transformer-classifier`
  артефактов в demo-стенд;
- хранение истории диалогов в PostgreSQL;
- расширение LLM-режима и сравнение качества ответов;
- добавление метрик качества retrieval и generation.
