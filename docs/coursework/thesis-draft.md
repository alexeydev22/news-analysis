# Черновик пояснительной записки

Этот файл содержит рабочий текст, который можно перенести в документ курсовой
работы и дополнить требованиями конкретной методички.

## Введение

Экономические новости ежедневно влияют на ожидания инвесторов, компаний,
потребителей и государственных органов. Рост ВВП, изменение инфляции, решения
центральных банков, динамика безработицы и торговые показатели могут менять
оценку рыночных рисков. При этом количество новостных сообщений велико, а их
ручной анализ требует времени и предметной подготовки.

Автоматические диалоговые системы позволяют упростить работу с такими данными:
пользователь формулирует вопрос естественным языком, а система находит
релевантные источники, анализирует их и формирует краткий ответ. Использование
языковых моделей и retrieval-подхода особенно полезно для задач, где ответ
должен опираться не только на общие знания модели, но и на конкретные найденные
документы.

Работа находится на стыке машинного обучения и backend-разработки. ML/NLP-часть
включает представление текста, retrieval, классификацию влияния новости,
подготовку размеченных датасетов, обучение нескольких режимов классификатора и
подключение языковой модели. Backend-часть обеспечивает микросервисную
архитектуру, асинхронное выполнение, загрузку CSV-датасетов через
веб-интерфейс, управление активным набором данных и воспроизводимый запуск.

Объектом работы являются экономические новости. Предметом работы являются
методы построения автоматической диалоговой системы для поиска, анализа и
обобщения экономических новостей.

Цель работы: разработать локальную автоматическую диалоговую систему, которая
принимает вопрос пользователя, находит релевантные экономические новости,
оценивает их влияние и формирует ответ на основе найденных источников.

Для достижения цели поставлены задачи:

1. Изучить предметную область анализа экономических новостей.
2. Определить ML/NLP pipeline: представление текста, retrieval, анализ влияния
   и генерацию ответа.
3. Спроектировать микросервисную архитектуру диалоговой системы.
4. Реализовать сервис загрузки и индексации новостей.
5. Реализовать сервис векторного поиска релевантных новостей.
6. Реализовать сервис анализа влияния новости.
7. Подготовить CLI-сценарий преобразования внешнего CSV и обучения
   классификаторов.
8. Реализовать сервис генерации ответа с поддержкой языковой модели.
9. Реализовать веб-интерфейс для работы пользователя.
10. Подготовить воспроизводимый сценарий запуска и проверки.

Практическая значимость работы заключается в создании демонстрационного
стенда, который показывает полный путь обработки вопроса: от загрузки новостей
и выбора активного датасета до ответа с источниками, оценкой влияния и
confidence score.

## 1. Анализ предметной области

Экономические новости представляют собой текстовые сообщения о событиях,
которые могут влиять на рынок, отрасли и отдельные компании. Такие события
часто связаны с макроэкономическими показателями, монетарной политикой,
торговлей, занятостью, промышленностью и потребительской активностью.

Для автоматического анализа новостей применяются методы обработки естественного
языка. В рамках данной работы используются три типа задач:

- поиск релевантных документов по вопросу пользователя;
- классификация влияния новости;
- генерация итогового ответа в диалоговом формате.

Поиск релевантных документов реализуется через retrieval pipeline. Новость
представляется в виде текстового документа, после чего преобразуется в векторное
представление и сохраняется в векторном хранилище. При вопросе пользователя
система получает похожие документы и использует их как контекст.

Анализ влияния новости реализуется как классификация на три класса:
`positive`, `negative` и `neutral`. Для стабильной локальной демонстрации
используется режим `tfidf-logreg`, так как он быстро работает и не требует
загрузки тяжелой модели. Для исследовательского сценария предусмотрена
подготовка размеченного CSV через `prepare_dataset`, обучение baseline-модели,
модели на embedding-признаках и легкого transformer-классификатора, а также
сравнение результатов.

Генерация ответа выполняется в отдельном `dialog-service`. Сервис поддерживает
детерминированный template-режим и LLM-режим через OpenAI-compatible локальный
сервер. В проекте выбран легкий вариант подключения языковой модели:
`Qwen3-0.6B-Instruct-GGUF` через `llama.cpp`. Такой подход позволяет сохранить
возможность запуска на обычном ноутбуке и при этом показать архитектуру системы
на основе языковой модели.

## 2. Проектирование системы

Система построена как локальный микросервисный стенд. Это позволяет разделить
ответственность компонентов и показать промышленный подход к проектированию,
не превращая курсовую работу в один монолитный скрипт.

Основные компоненты:

| Компонент | Ответственность |
| --- | --- |
| `frontend-web` | Веб-интерфейс чата, загрузки CSV-датасета, выбора активного набора и просмотра источников. |
| `api-gateway` | Единая точка входа, оркестрация поиска, анализа и генерации ответа. |
| `news-service` | Загрузка CSV, хранение активного датасета, предпросмотр и запуск индексации новостей. |
| `news-worker` | Фоновая индексация через Taskiq и Redis. |
| `retrieval-service` | Индексация документов и поиск релевантных новостей в Qdrant. |
| `analysis-service` | Классификация влияния новости. |
| `dialog-service` | Формирование итогового ответа. |
| `Qdrant` | Векторное хранилище. |
| `Redis` | Очередь фоновых задач и канал событий. |
| `MLflow` | Хранение артефактов и результатов ML-экспериментов. |

Backend-сервисы используют слоистую архитектуру:

- `domain` содержит доменные модели и ошибки;
- `application` содержит use cases и `Protocol`-интерфейсы портов;
- `infrastructure` содержит адаптеры к внешним системам;
- `presentation` содержит FastAPI routes и преобразование ошибок;
- `main` содержит настройки, dependency injection и сборку приложения.

Такое разделение делает систему поддерживаемой. Например, можно заменить
Qdrant на другое векторное хранилище, не меняя use case поиска, или заменить
template generator на LLM generator, не меняя контракт `dialog-service`.

## 3. Реализация

Проект реализован в формате monorepo. Backend написан на Python с использованием
FastAPI, asyncio и Granian. Для dependency injection используется Dishka. Для
типизированных контрактов используется Pydantic. Межсервисные HTTP-вызовы
выполняются через Zapros. Структурные логи формируются через structlog.

Фоновая обработка вынесена в `news-worker`. Задачи ставятся через Taskiq, Redis
используется как backend очереди, а FastStream применяется для публикации
событий индексации. Такой вариант легче RabbitMQ и лучше подходит для локального
учебного стенда.

`news-service` поддерживает два источника данных. Если пользователь не загружал
CSV, используется демонстрационный файл `data/raw/economic_news.csv`. Если CSV
загружен через API или интерфейс, сведения о нем сохраняются в каталоге
`data/uploads`, а выбранный файл фиксируется как активный датасет в
`active_dataset.json`. После этого предпросмотр и индексация работают с
активным загруженным набором данных.

Векторный поиск реализован в `retrieval-service`. Для хранения векторов выбран
Qdrant. В demo-режиме возможно использование статических embeddings, чтобы
запуск в Docker Compose не зависел от скачивания модели.

Анализ новости реализован в `analysis-service`. Основной демонстрационный режим
`tfidf-logreg` возвращает класс влияния, confidence score и объяснение на
русском языке. В контрактах также предусмотрены режимы `embedding-logreg` и
`tiny-transformer-classifier`, которые можно подключать через ML-артефакты.

Для подготовки ML-артефактов реализован CLI-сценарий `prepare_dataset`.
Например, команда `just prepare-dataset path/to/external.csv --label-column
sentiment` приводит внешний CSV к схемам приложения и обучения. После этого
команды `just train-baseline`, `just train-embedding`, `just train-transformer`
и `just compare-models` обучают и сравнивают классификаторы, а
`just demo-up-trained` запускает стенд с обученными артефактами.

Диалоговый сценарий реализован через `api-gateway` и SSE. Пользователь видит не
только финальный ответ, но и этапы выполнения: старт чата, поиск источников,
анализ, генерация ответа и завершение. Это удобно для защиты, потому что
демонстрирует внутренний pipeline системы.

Frontend реализован на React. Интерфейс позволяет выбрать модель анализа,
указать лимит источников, загрузить CSV, увидеть активный датасет, выполнить
предпросмотр, запустить индексацию и задать вопрос в чате. Ответ выводится
вместе с источниками, оценкой влияния и confidence score.

## 4. Тестирование и результаты

Для проверки проекта подготовлены unit-тесты, тесты контрактов, API-тесты,
frontend-тесты и demo smoke-сценарий. Smoke-сценарий запускается командой:

```bash
just demo-smoke
```

Он проверяет:

- доступность `api-gateway`;
- доступность `news-service`;
- чтение demo CSV;
- индексацию новостей;
- постановку фоновой задачи;
- получение SSE-событий чата;
- доступность frontend HTML.

Ручной сценарий демонстрации:

1. Запустить стенд командой `just demo-up`.
2. Открыть `http://localhost:5173`.
3. При необходимости загрузить CSV и убедиться, что он отображается как
   активный датасет.
4. Нажать `Предпросмотр CSV`.
5. Нажать `Индексировать CSV`.
6. Задать вопрос: «Что означает рост ВВП и снижение инфляции для рынка?».
7. Проверить ответ, источники, confidence score и ход обработки.

Для проверки обученных режимов анализа используется отдельная последовательность
команд: подготовить CSV через `just prepare-dataset` с параметром
`--label-column`, выполнить `just train-baseline`, `just train-embedding`,
`just train-transformer`, затем сравнить результаты командой
`just compare-models` и запустить стенд командой `just demo-up-trained`.
После запуска необходимо последовательно выбрать все три режима анализа в UI и
убедиться, что каждый режим возвращает источники, класс влияния и confidence
score.

В результате система формирует ответ по найденным экономическим новостям,
показывает список источников и выводит оценку влияния каждой новости. Это
подтверждает, что реализован полный цикл автоматической диалоговой системы для
анализа экономических новостей.

## Заключение

В ходе работы была разработана локальная микросервисная диалоговая система для
анализа экономических новостей. Система принимает вопрос пользователя, ищет
релевантные новости, классифицирует их влияние и формирует ответ с указанием
использованных источников.

В проекте реализованы backend-сервисы, frontend-интерфейс, общий пакет
контрактов, общий framework-пакет, фоновая обработка задач, векторное хранилище
и воспроизводимый Docker Compose стенд. Архитектура использует слоистый подход,
DDD-границы, `Protocol`-интерфейсы и dependency injection.

Результат объединяет ML/NLP и backend-реализацию. ML/NLP-часть включает
подготовку датасета, обучение классификаторов, retrieval pipeline,
классификацию влияния, confidence score, объяснение результата, LLM-адаптер и
MLflow. Backend-часть включает микросервисы, DDD, асинхронные API, загрузку и
выбор активного CSV, фоновые задачи, SSE и React UI.

Поставленная цель достигнута. Проект соответствует теме курсовой работы,
поскольку реализует автоматический диалоговый сценарий на основе найденного
контекста, анализа экономических новостей и подключаемой языковой модели.

Дальнейшее развитие системы может включать подключение реальных новостных API,
расширение обучающих датасетов, обучение дополнительных моделей анализа,
хранение истории диалогов, сравнение качества LLM-ответов и расширение метрик
оценки retrieval pipeline.
