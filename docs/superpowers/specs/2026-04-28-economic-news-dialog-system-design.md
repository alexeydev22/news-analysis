# Design: локальная диалоговая система для анализа экономических новостей

Дата: 2026-04-28

## 1. Цель

Курсовой проект разрабатывается по теме: «Разработка автоматической диалоговой системы на основе языковой модели для анализа экономических новостей».

Цель проекта: создать локальную микросервисную диалоговую систему, которая анализирует экономические новости, оценивает их ожидаемое экономическое влияние, ищет релевантные материалы и формирует ответ пользователю на основе найденного контекста и результатов ML-моделей.

Проект должен одновременно закрывать две линии:

- исследовательскую: датасет, EDA, минимум 3 модели, метрики, гиперпараметры, сравнение и выводы;
- инженерную: DDD, микросервисы, FastAPI backend, React UI, локальная LLM, RAG-пайплайн, воспроизводимый запуск.

## 2. Основная ML-задача

Главная задача: классификация ожидаемого экономического влияния новости:

- `positive`;
- `neutral`;
- `negative`.

Эта постановка сильнее обычной тональности, потому что связывает NLP с экономическим смыслом: система оценивает не эмоциональную окраску текста, а возможное влияние новости на компанию, сектор, рынок или макроэкономический фон.

Дополнительные признаки и задачи:

- тематическая категория новости;
- ключевые компании, страны, секторы и финансовые инструменты;
- краткое объяснение факторов влияния в диалоговом ответе.

Главная метрика и сравнение моделей строятся вокруг классификации экономического влияния, чтобы курсовая не распалась на несколько независимых исследований.

## 3. Данные

Используется гибридный подход:

- готовый размеченный датасет для обучения и оценки моделей;
- небольшой корпус свежих экономических новостей для демонстрации RAG и диалогового сценария.

Обязательные этапы работы с данными:

- описание источников и структуры датасетов;
- проверка пропусков и дублей;
- распределение классов;
- распределение длин текстов;
- примеры новостей и разметки;
- train/validation/test split с обоснованием;
- сохранение подготовленных данных в `data/processed`.

## 4. Модели и эксперименты

В исследовательской части сравниваются минимум 3 варианта:

1. `tfidf-logreg` или `tfidf-svm`: быстрый baseline.
2. `rubert-tiny2-classifier`: легкая transformer-модель через эмбеддинги и классификатор.
3. `rubert-tiny2-finetuned` или компактная FinBERT-style модель: дообученная модель для основной задачи.

Модели должны переключаться в чате через UI и API-параметр `analysis_model`. Это связывает исследование с демонстрационной системой: пользователь видит, как меняется анализ одной новости при выборе разных моделей.

Метрики:

- accuracy;
- precision, recall, F1 macro;
- confusion matrix;
- время инференса;
- learning curves для моделей, где есть обучение;
- сравнение train/validation/test для проверки переобучения.

Подбор гиперпараметров:

- минимум 3 гиперпараметра;
- хотя бы один подбор в автоматическом режиме через grid search или аналогичный подход.

Experiment tracking:

- `MLflow` для запусков, метрик, параметров и артефактов;
- локальный volume для моделей, графиков, таблиц и индексов.

## 5. Диалоговая система

Диалоговый сценарий:

1. Пользователь задает вопрос об экономической новости, компании, секторе или событии.
2. `api-gateway` принимает запрос и передает его в нужные сервисы.
3. `retrieval-service` ищет релевантные новости в Qdrant.
4. `analysis-service` оценивает экономическое влияние найденных новостей выбранной моделью.
5. `dialog-service` формирует ответ через локальную LLM, используя найденные новости и результаты классификации.
6. UI показывает ответ, источники, выбранную модель, найденные новости и оценку влияния.

Для streaming-ответов используется SSE.

Локальная LLM:

- default: `Qwen3-0.6B-Instruct GGUF` через `llama.cpp`;
- рекомендуемый профиль: `Q8_0`, если укладывается в ресурсы;
- fallback: `Q6_K` или `Q4_K_M` для слабого железа.

LLM не должна самостоятельно выполнять весь экономический анализ. Ее роль: формировать связный ответ на основе RAG-контекста и результатов классификаторов.

## 6. Архитектура

Архитектура: микросервисная, слоистая, DDD-oriented.

Сервисы:

- `api-gateway`: внешний API, OpenAPI, SSE, агрегация ответов;
- `news-service`: загрузка датасетов и свежих новостей, нормализация, хранение метаданных;
- `analysis-service`: переключаемые классификаторы, оценка влияния, batch-анализ;
- `retrieval-service`: эмбеддинги, Qdrant, семантический поиск;
- `dialog-service`: prompt templates, вызовы `llama.cpp`, генерация ответа;
- `experiment-service`: MLflow-запуски, метрики, сравнение моделей, экспорт таблиц.

В каждом сервисе используется одинаковая структура:

```text
src/<service_name>/
  domain/
  application/
  infrastructure/
  presentation/
  main/
  workers/
tests/
```

Слои:

- `domain`: entities, value objects, domain errors, domain services;
- `application`: use cases, commands/queries, interfaces as `typing.Protocol`;
- `infrastructure`: реализации портов, DB, Redis, Qdrant, MLflow, llama.cpp, Zapros clients;
- `presentation`: FastAPI routers, SSE endpoints, request/response mapping;
- `main`: app factory, Granian entrypoint, Dishka providers;
- `workers`: Taskiq tasks и FastStream subscribers, только если они нужны сервису.

Принцип реализации: современная архитектура без кода ради кода. Не добавлять generic repository, abstract factory, manager/helper слои и события, если они не решают конкретную задачу.

## 7. Общие пакеты

Monorepo содержит:

```text
packages/
  framework/
  contracts/
```

`packages/framework` — внутренний технический foundation:

- base settings;
- настройка `structlog`;
- FastAPI app factory helpers;
- Dishka integration helpers;
- healthcheck helpers;
- error response helpers;
- lifecycle utilities для Redis/PostgreSQL;
- общие result/pagination/problem-details типы, если они реально используются.

`packages/framework` не содержит:

- domain entities;
- business use cases;
- model-specific logic;
- service-specific repositories;
- абстракции “на будущее”.

`packages/contracts` содержит:

- DTO;
- event schemas;
- API contracts;
- общие enum-типы вроде `ImpactLabel` и `AnalysisModelName`.

## 11. Структура презентации

Презентация рассчитана на 5 минут и 10 слайдов:

```text
1. Титульный слайд
2. Актуальность: почему экономические новости требуют автоматического анализа
3. Цель и задачи исследования
4. Данные: источники, классы, примеры новостей
5. EDA и подготовка: ключевые статистики и визуализации
6. Модели: 3 подхода и схема экспериментов
7. Результаты: таблица метрик + лучшая модель
8. Архитектура системы: микросервисы, DDD, RAG, локальная LLM
9. Демонстрационный сценарий: вопрос -> найденные новости -> оценка влияния -> ответ
10. Заключение, ограничения, источники
```

Распределение времени:

```text
Титул: 10 сек
Актуальность + цель: 45 сек
Данные и EDA: 55 сек
Модели и эксперименты: 90 сек
Результаты: 60 сек
Архитектура и демо: 60 сек
Заключение и источники: 40 сек
```

Презентация должна больше времени отдавать методам и результатам, а не перечислению backend-библиотек.

## 13. Acceptance Criteria

Дизайн считается успешным, если:

- структура проекта поддерживает микросервисную DDD-архитектуру;
- исследовательская часть явно закрывает требования методички;
- минимум 3 модели можно сравнить и переключать в чате;
- локальная LLM используется как генератор ответа поверх найденного и классифицированного контекста;
- презентация соответствует критериям оценки и укладывается в 5 минут;
- реализация остается поддерживаемой: архитектура сильная, но без слоев и абстракций ради самих слоев.
